# Seoullo 아키텍처

이 문서는 최종 구현 기준의 데이터 흐름, 검색 인덱스, 감정 추천과 동기화 정책을 설명합니다.

## 전체 구성

```text
Vue SPA (Netlify)
  ├─ Kakao Maps JavaScript SDK
  └─ HTTPS /api
        ↓
FastAPI (Render)
  ├─ SQLite: 장소·리뷰·태그·감정 프로필의 기준 데이터
  ├─ Pinecone: lexical 검색과 emotion 유사도 검색
  ├─ Kakao Local API: 주소 검색·역지오코딩
  └─ OpenAI API: 챗봇 답변, 추천 이유, 사용자 장소 감정 초깃값
```

SQLite를 기준 데이터 저장소로 사용하고 Pinecone은 검색용 파생 인덱스로 사용합니다. Pinecone 결과의 장소 ID는 반드시 SQLite에서 다시 조회해 실제 응답을 구성합니다.

## 데이터 초기화와 주소 보강

FastAPI가 시작되면 다음 순서로 준비합니다.

1. SQLAlchemy 모델 기준으로 테이블을 생성하고 기존 DB 호환 처리를 수행합니다.
2. `data/서울/서울_*.json`을 `content_id` 기준으로 중복 안전하게 적재합니다.
3. 여행코스(`content_type_id=25`) 중 주소가 비어 있는 장소를 Kakao 역지오코딩합니다.
4. 구한 주소와 우편번호를 `places`에 저장하고 `address_source=kakao_reverse`로 기록합니다.

역지오코딩은 동시 요청을 제한하고 실패 항목을 건너뜁니다. 원본 JSON은 어떤 경우에도 수정하지 않습니다. 영속 디스크가 없는 배포 환경에서는 재시작 후 새 SQLite DB에 같은 초기화가 수행됩니다.

## 장소 데이터 정책

| 구분 | 데이터셋 장소 | 사용자 장소 |
|---|---|---|
| 생성 | 서버 초기 적재 | 익명 사용자 양식 |
| 수정·삭제 | 불가 | 등록 비밀번호 필요 |
| 이미지 | 원본 URL 사용 | SQLite BLOB, 파일당 5MB 이하 |
| 태그 추가 | 누구나 가능 | 누구나 가능 |
| 태그 삭제 | 불가 | 장소 비밀번호 필요 |
| 감정 초깃값 | JSON `emotion` | OpenAI 구조화 응답의 16개 수치 |

장소당 태그는 최대 10개이며 각 태그는 6자 이하입니다. 이미지가 없으면 프론트엔드가 플레이스홀더를 표시합니다.

## Pinecone 인덱스

| 인덱스 기본명 | 형식 | 메트릭 | 용도 |
|---|---|---|---|
| `seoullo-lexical` | sparse | dotproduct | 제목·유형·태그·주소·설명 검색 |
| `seoullo-emotions` | dense 16차원 | cosine | 감정 기반 추천과 챗봇 감정 질의 |

### Lexical 인덱스

- 기본 네임스페이스: `places`
- 벡터 ID: `place:{place_id}`
- 제목 5, 유형 4, 태그 3, 주소 2, 설명 1의 필드 가중치를 적용합니다.
- 한국어 토큰과 문자 n-gram을 BLAKE2s로 해싱한 sparse 벡터를 사용합니다.
- 챗봇의 기본 장소 검색 경로로 사용합니다.

### Emotion 인덱스

- 기본 네임스페이스: `profiles`
- 벡터 ID: `emotion:{place_id}`
- 순서는 현재 기분 6개, 원하는 감정 6개, 여행 스타일 4개로 고정됩니다.
- 각 그룹을 L2 정규화한 뒤 `mood 0.40`, `afterFeeling 0.35`, `style 0.25` 가중치를 적용합니다.

## 감정 추천과 체크인

감정 추천 요청은 현재 기분을 하나 이상, 원하는 감정을 하나 이상, 여행 스타일을 정확히 하나 받습니다. 선택값을 16차원 질의 벡터로 만든 뒤 Pinecone cosine 유사도 상위 장소를 구하고 최종 5곳을 반환합니다. Pinecone을 사용할 수 없으면 SQLite의 같은 16차원 계산으로 대체합니다.

OpenAI는 순위 계산에 관여하지 않고 최종 추천 이유를 작성합니다. 호출 실패 시 규칙 기반 이유를 반환합니다.

체크인은 방문 장소, 방문 전 감정·강도, 방문 후 감정·강도, 여행 스타일을 저장합니다. 선택된 `mood`, `afterFeeling`, `style` 컬럼을 각각 1씩 증가시키며 강도는 이력으로만 보관합니다. 갱신된 16차원 프로필은 즉시 emotion 인덱스에 upsert합니다. 만족도 컬럼은 존재하지 않습니다.

## 챗봇 검색 흐름

1. GPT-5 mini와 규칙을 조합해 질의 의도·검색어·필터를 정리합니다.
2. 일반 장소 질의는 lexical 인덱스, 감정 질의는 emotion 인덱스를 우선 사용합니다.
3. 두 검색이 모두 유효하면 reciprocal rank fusion으로 결합합니다.
4. Pinecone 장애나 결과 부족 시 SQLite 키워드·감정·인기순 검색으로 대체합니다.
5. 검색된 ID를 SQLite 장소 데이터로 검증한 뒤 최대 3개 카드와 답변을 생성합니다.

사용자 응답에는 내부 `place_id`, 인덱스명, RAG·문서 같은 구현 용어를 노출하지 않습니다.

## 변경 시 인덱스 동기화

| 변경 | Lexical | Emotion |
|---|---:|---:|
| 사용자 장소 생성 | upsert | upsert |
| 사용자 장소 수정 | upsert | 기본값 재생성 후 upsert |
| 사용자 장소 삭제 | delete | delete |
| 태그 추가·삭제 | upsert | - |
| 감정 체크인 | - | upsert |

장소 생성·수정, 태그 변경, 체크인은 Pinecone 동기화가 실패하면 관련 SQLite 트랜잭션을 롤백하고 503을 반환합니다. 장소 삭제는 벡터 삭제가 성공한 뒤 SQLite 행을 삭제합니다. 운영 복구가 필요하면 `rebuild_pinecone` 관리 명령으로 인덱스를 다시 구성합니다.

## 익명 식별과 비밀번호

리뷰·좋아요 중복 방지는 요청 IP와 User-Agent를 서버 비밀키로 HMAC 처리한 식별값을 사용합니다. 원본 IP와 User-Agent는 저장하지 않습니다. 장소와 리뷰의 수정·삭제 비밀번호는 프로젝트 요구사항에 따라 평문 저장·비교하므로 실사용 계정 비밀번호를 입력하지 않도록 안내해야 합니다.
